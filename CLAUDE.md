# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

RAPP — Automotive AI Repair Engine. Converts VIN + symptoms/OBD-II codes into tool-constrained, RAG-verified repair instructions. FastAPI backend (`backend/`), Next.js 14 App Router frontend (`frontend/`, TypeScript, pnpm, vanilla CSS — no Tailwind). Specs live in `ORIGINAL_REQUEST.md`, `PHASE_1_SPEC.md`, and `product_evolution_spec.md`; acceptance criteria in `ORIGINAL_REQUEST.md` are the source of truth for what must keep working.

## Commands

The Makefile assumes Poetry, but **Poetry is not installed on this machine**. Use the repo-root `.venv` directly for Python and `./node_modules/.bin/` for Next.js:

```bash
# Backend dev server (port 8000)
./.venv/bin/python3 -m uvicorn backend.main:app --reload --host 127.0.0.1 --port 8000

# Frontend dev server (port 3000) — run from frontend/
./node_modules/.bin/next dev -p 3000

# Frontend production build (must pass with zero TS/ESLint errors)
cd frontend && ./node_modules/.bin/next build

# Backend unit tests (all / single)
./.venv/bin/pytest tests/unit/ -v
./.venv/bin/pytest tests/unit/test_api.py::test_health_check -v

# E2E fault-injection harness (the canonical "did I break the frontend" check)
./tests/verify_tests.sh

# Playwright directly against a running frontend (single browser is much faster)
FRONTEND_URL=http://localhost:3000 npx playwright test --project=chromium
```

pnpm quirk: `pnpm build` / `pnpm dev` run a dependency check that fails with `ERR_PNPM_IGNORED_BUILDS` (protobufjs et al.), and edits to `frontend/pnpm-workspace.yaml` get reverted by something outside normal tooling. Invoking `./node_modules/.bin/next` directly bypasses this entirely. Also: never `rm -rf frontend/.next` while a dev server is running — it corrupts the running server's webpack chunks ("Cannot find module './NNN.js'"); kill the server first.

Lint/type-check (backend): `ruff check backend/`, `black --check backend/`, `mypy backend/` (strict mode; `tests/` and `frontend/` are excluded). CI (`.github/workflows/ci.yml`) runs all three plus unit tests, then Playwright against the mock app.

## Architecture

**Backend** — all routes intentionally flat in a single `backend/main.py` (per spec; don't split into routers). Endpoints: `/health`, `GET /api/vin/{vin}`, `POST /api/diagnose`, `POST /api/repair`, `POST /api/payments/*`. Config via pydantic-settings, structlog logging, centralized exception handler that never leaks stack traces.

- **VIN decoding** supports two schemes: real 17-char VINs via the live NHTSA API, and synthetic `SYN{yy}{MAKE}{MODEL}` VINs generated by the frontend's Year/Make/Model dropdown (hardcoded map of 5 makes in `decode_vin_internal`). The live NHTSA API is flaky for some fixture VINs (returns `Model: None` for `1HGBH41JXMN109186`) — E2E failures on the VIN-text-entry path are often this, not your code.
- **RAG isolation rule**: nothing outside `backend/rag/` may import `chromadb`. This is enforced by `tests/unit/test_rag.py::test_import_hygiene`. `backend/rag/__init__.py` is a thread-safe singleton factory selecting the backend via `VECTOR_STORE` env var (`chromadb` | `mock`); `retriever.py` exposes `retrieve(query, vin_meta, k=5)`.
- **No server-side auth or sessions.** The paywall "unlock" is just a Stripe session id the frontend stores in localStorage; `/api/repair` returns 402 without one. There is deliberately no login route or auth.py. Firebase auth (below) is entirely client-side.

**Frontend** — 4-step flow: `/` (VIN entry: text, YMM cascade, or Tesseract.js OCR photo scan) → `/diagnose` → `/results` (free summary + paywall) → `/repair` (unlocked guide). Plus `/repair/success` (Stripe redirect handler) and `/garage` (saved repairs).

- **All API calls go through `frontend/src/lib/api.ts`** — the single typed fetch wrapper. Don't add raw `fetch` calls.
- **localStorage is the session layer.** Keys are load-bearing and asserted by tests: `rapp_vin`, `rapp_vin_data` (decoded vehicle JSON), `rapp_symptoms`, `rapp_tools`, `rapp_unlocked_{vin}` (set by `/repair/success`, checked by `/repair`), `rapp_garage_dismissed_{vin}`. Pages redirect to `/` when required keys are missing, so tests pre-populate them via `page.evaluate`.
- **Repair page phase structure**: `/api/repair` returns a flat `string[]` of steps; the page buckets them client-side into Phases 1–4 via quartile math (`p1/p2/p3` in `repair/page.tsx`). Phase 5 (Conclusion & Verification) is deliberately *not* sliced from API steps — it's the static/templated `ConclusionPhase.tsx`, because the backend never produces OBD-clearing/test-drive content. Inline diagrams (`repair/diagrams/`) are placed by regex heuristics over step text (torque → LayoutDiagram in Phase 3, wiring/connector → WiringDiagram), rendered *inside* the step `<li>` so index-based `checkedSteps` state stays aligned.
- **Chat panel** (`repair/ChatPanel.tsx`) is a persistent side column on desktop (two-column `.repair-shell` layout), bottom drawer under 900px. Replies are intentionally mock/keyword-matched. Its contextual opening message recomputes as `vinData`/`symptoms` arrive async — don't turn that effect back into a run-once mount effect.
- **Firebase** (`src/lib/firebase.ts`, `auth.ts`, `repairs.ts`) is client-only, v9 modular API, and must **no-op gracefully when `NEXT_PUBLIC_FIREBASE_*` env vars are unset** (getters return `null`; UI shows "not configured" states). No real Firebase project exists yet — don't make anything hard-depend on it. Firestore schema: `users/{uid}` + `users/{uid}/repairs/{repairId}`.
- **CSS**: extend `globals.css`, don't replace it. Design tokens are CSS variables (`--bg-primary`, `--accent-orange`, etc.). The Tailwind-looking classes (`.dark`, `.bg-slate-900`, `.border-orange-500`, `.bg-orange-950`, `.text-orange-500`) are **not Tailwind** — they're hand-written hooks that Playwright asserts. `<body className="dark bg-slate-900">` in `layout.tsx` is asserted by tests. Manufacturer logos come from `logo.clearbit.com` via `src/lib/logos.ts` with a mandatory inline-SVG fallback (the CDN is unofficial and fails offline; never gate UI on it).

## Test infrastructure (unusual — read before touching tests)

`tests/e2e-mvp-flow.spec.ts` selectors are **frozen contracts**. Never rename/remove these `data-testid`s: `vin-input`, `submit-vin-btn`, `scan-barcode-btn`, `select-year/make/model`, `submit-ymm-btn`, `symptoms-input`, `tool-hand-tools`, `tool-jack-stands`, `tool-multimeter`, `submit-diagnose-btn`, `free-diagnosis-summary`, `locked-repair-steps`, `payment-cta-btn` (must contain "Unlock"), `safety-warning-banner` (orange classes, no dismiss button), `detailed-repair-steps`, `rag-citation`, `back-to-home-btn`. UI requirements the tests enforce: ≥48px touch targets on all interactive elements, dark-mode body class.

`tests/mock_app.py` is a self-contained FastAPI app that mimics the *entire* frontend+backend on one port (default 3099) purely to validate the E2E suite itself. `tests/verify_tests.sh` runs the suite 5 times against it — once healthy (must pass) and once per fault-injection env flag (`FAULTY_VIN_DECODING`, `MISSING_WARNINGS`, `BYPASS_PAYWALL_GATE`, `SMALL_TOUCH_TARGETS` — each must make the suite fail). So verify_tests.sh passing means the *tests* are sound, not the real app; to test the real app, run Playwright with `FRONTEND_URL` pointing at the real Next.js server with the real backend on 8000.

Playwright defaults to `http://localhost:3099` (mock port) and 5 browser projects; use `--project=chromium` and `FRONTEND_URL` for quick real-app runs.
