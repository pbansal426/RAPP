# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

RAPP — Automotive AI Repair Engine. Converts VIN + symptoms/OBD-II codes into tool-constrained, RAG-verified repair instructions. FastAPI backend (`backend/`), Next.js 14 App Router frontend (`frontend/`, TypeScript, pnpm, vanilla CSS — no Tailwind). Specs live in `ORIGINAL_REQUEST.md`, `PHASE_1_SPEC.md`, and `product_evolution_spec.md`; acceptance criteria in `ORIGINAL_REQUEST.md` are the source of truth for what must keep working.

## Commands

Backend dependency management is `uv` (Astral) — no more Poetry, no more manually-managed `.venv`. `uv run <cmd>` resolves the lockfile (`uv.lock`) and executes inside the project's venv on the fly; `uv sync --all-groups` installs/updates it explicitly. Use `./node_modules/.bin/` for Next.js:

```bash
# Backend dev server (port 8000)
uv run uvicorn backend.main:app --reload --host 127.0.0.1 --port 8000

# Frontend dev server (port 3000) — run from frontend/
./node_modules/.bin/next dev -p 3000

# Frontend production build (must pass with zero TS/ESLint errors)
cd frontend && ./node_modules/.bin/next build

# Backend unit tests (all / single)
uv run pytest tests/unit/ -v
uv run pytest tests/unit/test_api.py::test_health_check -v

# E2E fault-injection harness (the canonical "did I break the frontend" check)
./tests/verify_tests.sh

# Playwright directly against a running frontend (single browser is much faster)
FRONTEND_URL=http://localhost:3000 npx playwright test --project=chromium
```

pnpm quirk: `pnpm build` / `pnpm dev` run a dependency check that fails with `ERR_PNPM_IGNORED_BUILDS` (protobufjs et al.), and edits to `frontend/pnpm-workspace.yaml` get reverted by something outside normal tooling. Invoking `./node_modules/.bin/next` directly bypasses this entirely. Also: never `rm -rf frontend/.next` while a dev server is running — it corrupts the running server's webpack chunks ("Cannot find module './NNN.js'"); kill the server first.

Lint/type-check (backend): `uv run ruff check backend/`, `uv run black --check backend/`, `uv run mypy backend/` (strict mode; `tests/` and `frontend/` are excluded). CI (`.github/workflows/ci.yml`) runs all three plus unit tests, then Playwright against the mock app.

## Architecture

**Backend** — modularized into packages; `backend/main.py` is now a thin backward-compatible shim only (see below), not the app. This supersedes ORIGINAL_REQUEST.md's original "all routes flat in a single main.py" acceptance criterion, which was true for Phase 1 but became untenable once the file passed 1,000+ lines. Layout:
  - `backend/app.py` — the real FastAPI bootstrap: app instance, CORS, lifespan, exception-handler registration, `include_router` calls, `/health`.
  - `backend/core/` — `config.py` (pydantic-settings `Settings`), `logging.py` (structlog setup), `exceptions.py` (centralized handlers that never leak stack traces).
  - `backend/routers/` — `vin.py` (`GET /api/vin/{vin}`, `POST /api/vin/ocr`), `diagnose.py` (`POST /api/diagnose`), `repair.py` (`POST /api/repair`), `payments.py` (`POST /api/payments/*`).
  - `backend/services/` — `gemini.py` (the one place that touches `google-genai`: client construction, text/structured-output/vision calls), `rag.py` (thin pass-through to `backend/rag/`), `stripe.py` (mock checkout URL helpers today; the seam for a real Stripe SDK call later).
  - `backend/schemas.py` — every request/response Pydantic model (`VehicleInfo`, `DiagnoseRequest`, `DiagnoseResponse`, `RepairRequest`, `RepairResponse`, `CheckoutRequest`, `CheckoutResponse`, `VinOcrResponse`, etc.) shared across routers.
  - `backend/main.py` — re-exports `app`/`settings` and a handful of internals (`decode_vin_internal`, `_sanitize_vin_candidate`, `get_genai_client`, ...) so `uvicorn backend.main:app` / the Dockerfile's gunicorn CMD keep working unchanged. New code should import from the module that actually owns the symbol, not from this shim.

- **VIN decoding** supports two schemes: real 17-char VINs via the live NHTSA `DecodeVinValuesExtended` endpoint (retried up to 3x with exponential backoff via `tenacity`, `User-Agent` header set — see `_fetch_nhtsa_vin_fields`/`_extract_vehicle_data` in `backend/routers/vin.py`), and synthetic `SYN{yy}{MAKE}{MODEL}` VINs generated by the frontend's Year/Make/Model dropdown (hardcoded map of 5 makes in `decode_vin_internal`). The real-VIN path returns a richer payload now: `trim`, `body_class`, `vehicle_type`, `fuel_type`, and a derived `powertrain` (from NHTSA's `ElectrificationLevel`/`FuelTypePrimary` fields) in addition to `year`/`make`/`model`/`engine`/`drive_type`. An unambiguous decoded `powertrain` means the frontend shouldn't prompt the user to pick one manually. `engine` is a short human string like `"2.5L I4"` or `"3.5L V6"` (config abbreviated from NHTSA's `EngineConfiguration`). The live NHTSA API can still be flaky/slow for some fixture VINs — E2E failures on the VIN-text-entry path are often this, not your code. If NHTSA is unreachable after retries (5xx/403/timeout/connect error), `decode_vin_internal` falls back to `wmi_fallback_decode` in `backend/vin_fallback.py`, which decodes only `make` (from the 3-char WMI, covering Toyota/Honda/Ford/Chevrolet) and `year` (from the standardized position-10 code) from the VIN string itself — `model`/`engine`/`trim`/`drive_type` are deliberately left blank rather than guessed, since full VDS decoding is manufacturer-proprietary. This response additively includes `decode_source: "offline_fallback"` so callers can tell live-decoded data from best-effort data; a 502 is still raised when the WMI isn't in the fallback table.
- **`POST /api/vin/ocr`** accepts a `multipart/form-data` image (`file` field; JPEG/PNG/WEBP/HEIC, ≤20MB raw upload) of a VIN tag/door-jamb sticker/registration doc. `_normalize_image_for_vision` in `backend/routers/vin.py` always decodes the upload via Pillow and re-encodes as JPEG, downscaling first if the longest edge exceeds 2048px — real phone photos are routinely 8-15MB at full sensor resolution, far more than an embossed VIN plate needs to stay legible, so this keeps the Gemini request small regardless of source size (matters for both latency and the raw-upload cap). It then extracts the VIN via Gemini's native multimodal vision (`gemini-3.5-flash`, structured output via a `VinOcrExtraction` Pydantic schema in `backend/services/gemini.py` — no more raw JSON dict handling), sanitizes it (strips punctuation, uppercases, excludes I/O/Q), scores it against the standard VIN check-digit algorithm, and returns `{ vin, confidence, decoded_vehicle }` where `decoded_vehicle` is the same shape as `GET /api/vin/{vin}` (or `null` if the extracted VIN doesn't decode). Returns 503 if `GEMINI_API_KEY` isn't set, 422 if no VIN could be read from the photo, 429 if Gemini itself rejects the call for quota/rate-limit reasons (`GeminiRateLimitError` in `backend/services/gemini.py`, raised only for HTTP 429 from the `google-genai` SDK) — this is deliberately a distinct status from 422 so a caller (e.g. the frontend's continuous scan loop) doesn't mistake "Gemini is throttled" for "this photo has no readable VIN" and keep hammering a dead quota. Gemini's free tier is a mere 20 requests/day per model, which a live scan loop can exhaust in well under a minute; there is no server-side backoff/queueing here, so hitting it in practice means either waiting out the daily reset or moving to a paid Gemini tier. `OPENAI_API_KEY` no longer exists anywhere in the stack — Gemini is now the only LLM provider.
- **RAG isolation rule**: nothing outside `backend/rag/` may import `chromadb`. This is enforced by `tests/unit/test_rag.py::test_import_hygiene`. `backend/rag/__init__.py` is a thread-safe singleton factory selecting the backend via `VECTOR_STORE` env var (`chromadb` | `mock`); `retriever.py` exposes `retrieve(query, vin_meta, k=5)`. `backend/services/rag.py` is a thin pass-through routers call instead of reaching into `backend/rag/` directly — it still must not import `chromadb` itself.
- **Auth is real and server-side now** — `backend/routers/auth.py` (`POST /api/auth/signup`, `POST /api/auth/login`, `GET/PATCH /api/auth/me`, `POST /api/auth/forgot-password`, `POST /api/auth/reset-password`, `POST /api/auth/send-verification`, `POST /api/auth/verify-email`) and `backend/routers/repairs.py` (`POST /api/repairs`, `GET /api/repairs`). This supersedes ORIGINAL_REQUEST.md's R5 spec, which called for client-side Firebase Auth/Firestore — that was built once, then replaced with this backend-owned stack (see "Auth & account storage" below) because the app already runs a FastAPI server 24/7, so a second (Firebase) identity system alongside it was pure duplication with its own quota/vendor surface to track. The paywall "unlock" is still just a Stripe session id the frontend stores in localStorage; `/api/repair` returns 402 without one — that part is unrelated to user accounts and unchanged.
- **Auth & account storage**: `backend/core/models.py` defines `DbUser` (`users` table, includes `email_verified: bool`) and `DbSavedRepair` (`saved_repairs` table) via SQLAlchemy 2.0 `Mapped`/`mapped_column`. `backend/core/database.py` points at `DATABASE_URL` (defaults to `sqlite:///./data/rapp.db` for local dev; point at Postgres in prod) and creates tables via `Base.metadata.create_all` in `app.py`'s lifespan on every startup — so **a stale server process that predates a models change, or whose sqlite file got deleted/replaced out from under it, needs a restart (not a migration) to pick up the current schema**; symptom is a 500 "Internal server error" server-side and a `TypeError: Failed to fetch` / `net::ERR_FAILED` client-side, because FastAPI's default exception path doesn't attach CORS headers to that response and the browser reports the whole request as a network failure rather than surfacing the 500. `backend/core/security.py` hashes passwords with `hashlib.scrypt` (not bcrypt/argon2 — no extra native dependency) and issues HS256 JWTs signed with `settings.jwt_secret_key` (`JWT_SECRET_KEY` env var — **always override the local-dev default in any deployed environment**, and prefer a key ≥32 bytes; PyJWT will warn on shorter ones). Three token purposes share that secret (`create_access_token`/`create_reset_token`/`create_verify_token`), each stamped with a `type` claim (`access`/`reset`/`verify`) and decoded via `decode_token(token, expected_type=...)` — this scoping is load-bearing: without it a leaked short-lived reset/verify token could be replayed as a full session token. Password reset and email verification are **dev-mode only**: no email provider is wired up, so `forgot-password`/`send-verification` return the reset/verify link directly in the JSON response (`reset_link`/`verify_link`) for the frontend to render as a clickable link instead of emailing it. Swap in a real provider (Resend/Postmark/SendGrid/...) before this goes anywhere near real users, and stop returning those fields once you do. Frontend: `frontend/src/lib/auth.ts` (`signUp`/`logIn`/`logOut`/`updateAccount`/`forgotPassword`/`resetPassword`/`sendVerification`/`verifyEmail`/`useAuthUser()`) stores the bearer token in localStorage under `rapp_token` and attaches it via `frontend/src/lib/api.ts`'s `authHeader()`; `frontend/src/lib/repairs.ts` (`saveRepair`/`listRepairs`) calls `/api/repairs` with that token, no `uid` param needed since the backend derives identity from it. Dedicated pages: `/signin`, `/signup`, `/forgot-password`, `/reset-password` (reads `?token=`), `/verify-email` (reads `?token=`), `/settings` (display name, email-verification status, log out) — all live under `frontend/src/app/`, all `'use client'`, and any page reading a search param wraps its content in `<Suspense>` (Next.js App Router requirement) matching the existing pattern in `repair/success/page.tsx`. `/garage` redirects to `/signin?next=/garage` when logged out rather than embedding its own auth form.
- **Local data storage & the external SSD**: the `data/` directory is split by role, not symlinked wholesale. `data/` itself is a **real directory on the internal disk**; only `data/chroma_db` and `data/etl_workspace` are symlinks to an external SSD (`/Volumes/Extreme SSD/dev/RAPP_data/...`). Rationale: the knowledge base (`chroma_db`) and raw source PDFs (`etl_workspace`) are large-and-growing but **re-buildable** (chroma via `etl/import_kb.py`/`kb_export`, PDFs via re-download), so they stay off the near-full internal disk; `data/rapp.db` (user accounts + saved repairs) is **tiny (~1 MB) and irreplaceable**, so it lives on the internal disk as the authoritative copy. Consequence: **the server always starts and the app's non-KB features (auth, VIN decode, diagnose/repair via Gemini + templates) keep working with the SSD unplugged** — startup only touches `rapp.db` (`app.py` lifespan), and RAG *fails open* to empty results when `chroma_db` is unreachable (`get_vector_store()` returns a transient, uncached `MockVectorStore` on construction failure, and `retrieve()` wraps the query in try/except → `[]`), auto-recovering once the SSD is replugged (no restart needed). Don't reintroduce a whole-`data/`-directory symlink — it makes the SSD a single point of failure for the accounts DB and 500s RAG calls when unplugged. **Durability for `rapp.db`**: `backend/core/backup.py` is the single implementation (`backup_rapp_db()` — SQLite online-backup API, *never* a raw file copy of the live WAL DB; writes timestamped copies to `<ssd>/backups/`, keeps last 20, no-ops cleanly when the SSD is absent). It runs automatically on server startup (`backup_rapp_db_safe()` in the lifespan, wrapped so it never blocks/fails boot) and on demand via `make backup-db`, `scripts/backup_rapp_db.sh`, or double-clicking `scripts/backup_rapp_db.command` in Finder. Safe to run repeatedly. Override the destination with `RAPP_BACKUP_DIR`. `chroma_db` is deliberately *not* auto-snapshotted to the Mac (would consume the free space the split is protecting) — use the existing `etl/export_kb.py` when you want an off-SSD KB snapshot.

**Frontend** — 4-step flow: `/` (VIN entry: manual text, YMM cascade, Upload Photo, or Scan VIN) → `/diagnose` → `/results` (free summary + paywall) → `/repair` (unlocked guide). Plus `/repair/success` (Stripe redirect handler) and `/garage` (saved repairs).

- **VIN entry on `/`** has four paths, all converging on the same `rapp_vin`/`rapp_vin_data` localStorage keys: manual 17-char text input; the Year/Make/Model cascade; **Upload Photo** (`frontend/src/app/page.tsx`'s `processVinImage`, plain file input with no `capture` attribute so it opens the photo library) which POSTs to `/api/vin/ocr` and falls back to a manual crop step (`VinCropModal.tsx`) + free Tesseract.js OCR if the backend call fails for any reason; and **Scan VIN** (`ScanModeModal.tsx`), which opens a chooser between `WindshieldScanModal.tsx` and `DoorJambScanModal.tsx`. Windshield mode live-scans the rear camera and polls `/api/vin/ocr` on a fixed ~1.5s interval (`SCAN_INTERVAL_MS`) until a response has *both* a VIN and a `decoded_vehicle` — a text-only read that fails to decode is treated as not-yet-found and the loop keeps going, matching a barcode-scanner's always-on feel; it stops immediately (rather than retrying) on any non-422 error, since 422 alone means "no VIN visible yet." Door Jamb mode decodes a Code 39/128/93 barcode continuously via `@zxing/browser` entirely client-side — no Gemini call at all, since barcode decoding is exact symbol matching. `frontend/src/lib/useCameraStream.ts` is the shared getUserMedia/torch-toggle hook for the windshield modal; door-jamb mode lets `@zxing/browser`'s `decodeFromConstraints` own the stream lifecycle instead, since it already needs full control for continuous decoding. **Gemini's free tier caps out at 20 requests/day per model** — the interval throttling above exists specifically so one live scan session doesn't blow through that budget in seconds; see the `/api/vin/ocr` 429 behavior below.

- **All API calls go through `frontend/src/lib/api.ts`** — the single typed fetch wrapper. Don't add raw `fetch` calls.
- **localStorage is the session layer.** Keys are load-bearing and asserted by tests: `rapp_vin`, `rapp_vin_data` (decoded vehicle JSON), `rapp_symptoms`, `rapp_tools`, `rapp_unlocked_{vin}` (set by `/repair/success`, checked by `/repair`), `rapp_garage_dismissed_{vin}`. Pages redirect to `/` when required keys are missing, so tests pre-populate them via `page.evaluate`.
- **Repair page phase structure**: `/api/repair` returns a flat `string[]` of steps; the page buckets them client-side into Phases 1–4 via quartile math (`p1/p2/p3` in `repair/page.tsx`). Phase 5 (Conclusion & Verification) is deliberately *not* sliced from API steps — it's the static/templated `ConclusionPhase.tsx`, because the backend never produces OBD-clearing/test-drive content. Inline diagrams (`repair/diagrams/`) are placed by regex heuristics over step text (torque → LayoutDiagram in Phase 3, wiring/connector → WiringDiagram), rendered *inside* the step `<li>` so index-based `checkedSteps` state stays aligned.
- **Chat panel** (`repair/ChatPanel.tsx`) is a persistent side column on desktop (two-column `.repair-shell` layout), bottom drawer under 900px. Replies are intentionally mock/keyword-matched. Its contextual opening message recomputes as `vinData`/`symptoms` arrive async — don't turn that effect back into a run-once mount effect.
- **Spec auto-lock** (`src/lib/vehicleSpecs.ts`): the home-page selector's powertrain/engine/drive auto-lock is table-driven — `lookupVehicleSpecs(year, make, model, trim)` against a curated list of vehicles with unambiguous specs (NHTSA can't resolve specs per *trim*, so this is hand-maintained). Only fields present on a matched entry get locked; a miss keeps manual selectors enabled. `tests/mock_app.py` mirrors this with its own small `specTable` — keep the two consistent for any vehicle the E2E suite exercises (currently Corolla and 2014–16 Highlander XLE, asserted by Steps 7–8 in `e2e-mvp-flow.spec.ts`).
- **`src/lib/firebase.ts` is a dead stub** — kept only so `isFirebaseConfigured()` (still called by a couple of presentational components) returns `true` without a real Firebase project. There is no Firebase project, no `NEXT_PUBLIC_FIREBASE_*` env vars, and no Firestore usage anywhere in this app; account creation is backend-driven (see "Auth & account storage" above) and always available, not conditional on any external provider. Don't reintroduce a Firebase dependency here — if presentational code still branches on `isFirebaseConfigured()`, treat that branch as dead code to simplify away, not a real capability check.
- **CSS**: extend `globals.css`, don't replace it. Design tokens are CSS variables (`--bg-primary`, `--accent-orange`, etc.). The Tailwind-looking classes (`.dark`, `.bg-slate-900`, `.border-orange-500`, `.bg-orange-950`, `.text-orange-500`) are **not Tailwind** — they're hand-written hooks that Playwright asserts. `<body className="dark bg-slate-900">` in `layout.tsx` is asserted by tests. Manufacturer logos come from `logo.clearbit.com` via `src/lib/logos.ts` with a mandatory inline-SVG fallback (the CDN is unofficial and fails offline; never gate UI on it).

## Pinned contract — Claude (backend) / Gemini (frontend) split

Work is currently split: Claude owns `backend/` and `frontend/src/lib/*.ts` (the typed data/API layer — `api.ts`, `firebase.ts`, `auth.ts`, `repairs.ts`, `nhtsa.ts`, `obdCodes.ts`, `logos.ts`, `vehicleSafety.ts`, `pendingSave.ts`); Gemini owns presentational frontend (`frontend/src/app/**/*.tsx`, `globals.css`) — visuals, copy, icons, layout. Neither side may change the shapes below without updating this section and notifying the other. Presentational code may read new fields off these shapes but must not invent new request/response fields, rename `data-testid`s, or change localStorage keys.

All shapes below are exported from `frontend/src/lib/types.ts` (`VehicleInfo`, `DiagnoseRequest`, `RepairRequest`, `DiagnosisResponse`, `RepairResponse`, `RecommendedPart`, `PartOption`, `CostBreakdown`, `VinDecodeResponse`, `VinOcrResponse`, `RecallInfo`, `RecallsResponse`, `ComplaintComponentFrequency`, `ComplaintsSummaryResponse`) — import from there instead of re-typing these shapes inline.

**`POST /api/diagnose` / `POST /api/repair` request body** (`RepairRequest` adds `stripe_session_id: string` on top of this):
```ts
{
  vin: string;
  symptoms: string;
  obd_codes: string[];   // always send as an array
  tools: string[];       // tool-* ids from the checkbox list
  vehicle: {              // optional — send whatever's in localStorage['rapp_vin_data'] verbatim
    year?: string | number;
    make?: string;
    model?: string;
    trim?: string;
    engine?: string;
    drive_type?: string;
    body_class?: string;
    vehicle_type?: string;
    fuel_type?: string;
    powertrain?: string;  // "Gasoline" | "Diesel" | "Hybrid" | "Plug-in Hybrid" | "Electric (EV)"
  } | null;
}
```
When `vehicle.make` is present, the backend trusts it directly instead of re-decoding the VIN — this is what makes the Year/Make/Model selector work for makes outside the old synthetic-VIN 5-make map. `rapp_vin_data` in localStorage should always be forwarded as `vehicle` verbatim; don't reshape it. `trim`/`body_class`/`vehicle_type`/`fuel_type` are accepted but not currently read by the backend — they're there so the full decoded object round-trips without the frontend needing to strip fields.

**`GET /api/vin/{vin}` response**: same shape as `vehicle` above but with `year`/`make`/`model` guaranteed present (404 otherwise) — see `VinDecodeResponse` in `types.ts`.

**`POST /api/vin/ocr`** (`multipart/form-data`, field name `file`) **response**: `{ vin: string; confidence: number; decoded_vehicle: VinDecodeResponse | null }`.

**`POST /api/diagnose` response**:
```ts
{
  summary: string;
  is_high_risk: boolean;
  high_risk_system: string | null;
  warning_message: string | null;
  recommended_parts: Array<{
    part_name: string;
    options: Array<{               // always exactly 3: one per tier
      tier: "OEM" | "Aftermarket / Budget" | "Upgrade";
      brand: string;
      part_number: string | null;
      title: string;
      estimated_price: number;
      purchase_url: string;        // Amazon/AutoZone/RockAuto search link
      rationale: string;
    }>;
  }>;
  cost_breakdown: {
    dealership_cost_range: [number, number];
    independent_shop_range: [number, number];
    parts_total: number;           // sum of the "Aftermarket / Budget" tier across recommended_parts
    diy_total: number;             // $4.00 RAPP fee + parts_total
    estimated_labor_hours: number;
  } | null;
}
```
`recommended_parts`/`cost_breakdown` are empty/zeroed (not omitted) when no repair template matches the symptoms/OBD codes — `cost_breakdown` is still always a non-null object with `parts_total: 0`.

**`POST /api/repair` response**: `{ repair_steps: string[]; citations: string[] }` — still a flat array of plain strings (backed by Gemini, the curated template library in `backend/repair_templates.py`, or RAG, in that preference order; frontend can't tell which produced it and shouldn't try). Torque callouts are guaranteed to start with the literal word `"Torque "` so the existing highlight regex in `repair/page.tsx` keeps working.

**`POST /api/repair/chat`** (new): request `{ vin, vehicle?, symptoms, repair_steps: string[], message, stripe_session_id }` — `repair_steps` is the exact procedure array already shown on `/repair`, sent back so the reply stays grounded in what the user is actually looking at rather than a fresh, possibly-different RAG query. Response `{ reply: string | null }` — `reply: null` means Gemini is unavailable/failed/quota-exhausted for this call; the caller (`ChatPanel.tsx`) falls back to its own local canned reply rather than surfacing an error. 402s exactly like `/api/repair` without a `stripe_session_id`. Backed by `backend/services/llm.py::generate_chat_reply`, grounded via a strict system prompt (no invented specs beyond what's in `repair_steps`). Gemini's shared 20-req/day free tier and this app's margin both mean the frontend must cap real calls per session rather than firing one per message unconditionally — see `ChatPanel.tsx` for the current cap.

**`POST /api/payments/create-checkout`** request now also accepts `symptoms: string` (used only for Stripe metadata, not required). Response: `{ checkout_url: string; mode: "mock" | "live" }`. `mode` is new and load-bearing for the frontend: `"live"` means `checkout_url` is a real `checkout.stripe.com` page that requires a genuine full-page redirect (`window.location.href`, not `router.push`) since the customer has to actually complete payment there; `"mock"` is the pre-existing stub that just 303s straight back with no real payment step, which the SPA still shortcuts past (see `handlePay` in `results/page.tsx`). Controlled entirely by whether `STRIPE_SECRET_KEY` is configured (`backend/services/stripe.py`) — unset by default, so this remains free/mock until a real Stripe account is wired in. `POST /api/webhooks/stripe` (new) verifies the `Stripe-Signature` header against `STRIPE_WEBHOOK_SECRET` and 503s if that secret isn't configured; on `checkout.session.completed` it only logs the event (vin/user_id from metadata) as an audit trail — it deliberately does not generate or save a repair guide itself, since the frontend already triggers and caches that generation client-side, and duplicating it server-side would mean a second Gemini call per purchase.

**`GET /api/vehicle-safety/recalls?year=&make=&model=`** and **`GET /api/vehicle-safety/complaints-summary?year=&make=&model=`** (new): live NHTSA lookups, deliberately **not** ingested into the RAG vector store the way TSBs are — recall status changes over time, and a bulk-ingested snapshot could tell a user "no open recall" when one was just issued, which the TSB corpus's staleness tolerance doesn't share. Backed by `backend/services/nhtsa_safety.py`, which hits `api.nhtsa.gov` live behind a short in-memory TTL cache (1h for recalls, 24h for complaints — recalls refresh more eagerly since they're the safety-critical signal) and never raises: an NHTSA failure/timeout degrades to an empty result rather than a 5xx, so `/results` never breaks just because this third-party lookup timed out. Recalls response: `{ open_recalls: Array<{ campaign_number, component, summary, consequence, remedy, report_received_date }>, count: number }`. Complaints response: `{ total_complaints: number, top_components: Array<{ component: string, count: number }> }` (component frequency, capped to the top 5 — NHTSA's `components` field on a complaint is a comma-separated string, not a list, and a multi-component complaint counts toward each). Surfaced on `/results` (free, pre-paywall) since recalls/complaints don't depend on symptoms and are valuable regardless of what the user is diagnosing.

**Frozen `data-testid`s** — see Test infrastructure section below for the Playwright-asserted ones. Also currently in use and not to be renamed: `select-powertrain`, `engine-detail`, `select-trim`, `select-drive` (home-page Year/Make/Model/Trim selector fields, asserted by E2E Steps 7–8) and `parts-plan-total` (the live DIY-budget footer on the results-page parts dashboard).

## Test infrastructure (unusual — read before touching tests)

`tests/e2e-mvp-flow.spec.ts` selectors are **frozen contracts**. Never rename/remove these `data-testid`s: `vin-input`, `submit-vin-btn`, `scan-barcode-btn`, `select-year/make/model`, `submit-ymm-btn`, `symptoms-input`, `tool-hand-tools`, `tool-jack-stands`, `tool-multimeter`, `submit-diagnose-btn`, `free-diagnosis-summary`, `locked-repair-steps`, `payment-cta-btn` (must contain "Unlock"), `safety-warning-banner` (orange classes, no dismiss button), `detailed-repair-steps`, `rag-citation`, `back-to-home-btn`. UI requirements the tests enforce: ≥48px touch targets on all interactive elements, dark-mode body class.

`tests/mock_app.py` is a self-contained FastAPI app that mimics the *entire* frontend+backend on one port (default 3099) purely to validate the E2E suite itself. `tests/verify_tests.sh` runs the suite 5 times against it — once healthy (must pass) and once per fault-injection env flag (`FAULTY_VIN_DECODING`, `MISSING_WARNINGS`, `BYPASS_PAYWALL_GATE`, `SMALL_TOUCH_TARGETS` — each must make the suite fail). So verify_tests.sh passing means the *tests* are sound, not the real app; to test the real app, run Playwright with `FRONTEND_URL` pointing at the real Next.js server with the real backend on 8000.

Playwright defaults to `http://localhost:3099` (mock port) and 5 browser projects; use `--project=chromium` and `FRONTEND_URL` for quick real-app runs.
